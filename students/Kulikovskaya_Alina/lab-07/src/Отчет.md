## Ход выполнения работы

### 1. Write Model

**Агрегат:** Booking (Бронирование)

**Структура:**
- Нормализованные таблицы
- Инварианты
  - Нельзя забронировать занятый слот
  - Проверка пересечения времени бронирований
  - Валидация статусов перехода (pending → confirmed → cancelled)

---

### 2. Read Model

**Проекция:** BookingView

**Структура:**
- Денормализованная таблица booking_views с предвычисленными полями
- JOIN предзагруженные: данные пользователя и площадки в одной таблице

---

### 3. Event-Driven Sync

**События:**
- `BookingCreatedEvent` → создать BookingView
- `BookingConfirmedEvent` → обновить статус и payment_status
- `BookingCancelledEvent` → обновить статус и cancelled_at

**Код:**
```python 
class BookingProjection:

    def __init__(self, read_repo: IBookingReadRepository):
        self._repo = read_repo

    def on_booking_created(self, event: BookingCreatedEvent) -> None:
        """Создание read model при бронировании."""
        view = BookingReadModel(
            booking_id=event.booking_id,
            user_id=event.user_id,
            user_email=event.user_email,  # денормализация
            user_phone=event.user_phone,
            court_id=event.court_id,
            court_name=event.court_name,  # денормализация
            court_type=event.court_type,
            date=event.slot.date,
            start_time=event.slot.start_time,
            end_time=event.slot.end_time,
            status="pending_payment",
            status_display="Ожидает оплаты",
            total_amount=event.total_amount,
            created_at=event.timestamp,
            confirmed_at=None,
            cancelled_at=None,
            is_past=False
        )
        self._repo.save(view)

    def on_booking_confirmed(self, event: BookingConfirmedEvent) -> None:
        """Обновление при подтверждении оплаты."""
        self._repo.update_status(
            booking_id=event.booking_id,
            status="confirmed",
            status_display="Подтверждено",
            confirmed_at=event.timestamp,
            payment_status="success"
        )

    def on_booking_cancelled(self, event: BookingCancelledEvent) -> None:
        """Обновление при отмене."""
        self._repo.update_status(
            booking_id=event.booking_id,
            status="cancelled",
            status_display="Отменено",
            cancelled_at=event.timestamp
        )

```

## Вывод

В ходе работы была реализована архитектура CQRS для системы бронирования спортивных площадок, основанная на полном разделении Write Model и Read Model.
Write-модель отвечает за выполнение команд (создание, подтверждение, отмена бронирования), изменение состояния агрегатов и генерацию доменных событий. Реализованы инварианты: проверка занятости слотов, валидация временных пересечений, корректные переходы статусов.
Read-модель обеспечивает формирование денормализованных проекций для быстрых запросов. Все данные для отображения списка бронирований (email пользователя, название площадки, тип корта) хранятся в одной таблице, что исключает JOIN-ы при чтении.
Event-Driven синхронизация через шину событий обеспечивает eventual consistency между моделями. Обработчики событий асинхронно обновляют read models, что позволяет оптимизировать производительность: write-операции не блокируются обновлением проекций.
