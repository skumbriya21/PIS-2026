## Ход выполнения работы

### 1. Booking Service

**Bounded Context:** Управление бронированиями

**Ответственность:**
- Создание и управление бронированиями
- Проверка доступности слотов через Court Service (gRPC)
- Публикация событий бронирования

**API:**
- `POST /api/bookings` — создать бронирование
- `GET /api/bookings/{id}` — получить бронирование
- `POST /api/bookings/{id}/confirm` — подтвердить оплату
- `POST /api/bookings/{id}/cancel` — отменить бронирование
- `GET /api/users/{user_id}/bookings` — список бронирований пользователя

**Код:**
```python
# contexts/booking/src/api/rest/booking_controller.py
@app.post("/api/bookings", response_model=BookingResponse, status_code=201)
async def create_booking(request: CreateBookingRequest):
    booking_id = await booking_service.create_booking(
        user_id=request.user_id,
        court_id=request.court_id,
        slot_date=request.slot_date,
        slot_time=request.slot_time,
        amount=request.amount
    )
    return BookingResponse(booking_id=booking_id, status="pending_payment")
```

---

### 2. Court Service

**Bounded Context:** Управление площадками и расписанием

**Ответственность:**
- Управление спортивными площадками (CRUD)
- Управление расписанием и доступностью слотов
- gRPC API для внутренней коммуникации

**API:**
- `GET /api/courts` — список площадок
- `GET /api/courts/{id}` — информация о площадке
- `GET /api/courts/{id}/schedule?date=YYYY-MM-DD` — расписание на дату
- gRPC: `CheckAvailability`, `ReserveSlot`, `ReleaseSlot`

---

### 3. Payment Service

**Bounded Context:** Сервис получения данных

**Ответственность:**
- Создание платежей
- Обработка вебхуков от платёжной системы
- Возвраты средств при отмене
- Публикация событий оплаты

**API:**
- `POST /api/payments` — создать платёж
- `GET /api/payments/{id}` — статус платежа
- `POST /api/payments/webhook` — вебхук от платёжной системы
- `POST /api/payments/{id}/refund` - возврат средств

---

### 4. Notification Service

**Bounded Context:** Уведомления пользователям

**Ответственность:**
- Отправка email и SMS
- Подписка на события других сервисов
- Напоминания о предстоящих бронированиях

---

### 5. Event Bus (RabbitMQ)

**События между сервисами:**
| Событие | Издатель | Подписчики |
|----------|-------|-----------|
| BookingCreatedEvent | Booking Service | Payment, Notification |
| BookingConfirmedEvent | Booking Service | Notification |
| BookingCancelledEvent | Booking Service | Payment (возврат), Notification |
| PaymentReceivedEvent | Booking Service | Booking (подтверждение) |
| PaymentFailedEvent | Booking Service | Booking (отмена), Notification |
| PaymentInitiatedEvent | Booking Service | Notification |

---

### 5. API Gateway

**Маршрутизация запросов:**
| Путь | Сервис | Описание |
|----------|-------|-----------|
| /api/bookings/** | Booking Service | Управление бронированиями |
| /api/courts/** | Court Service | Площадки и расписание |
| /api/payments/** | Payment Service | Платежи |
| /api/notifications/** | Notification Service | Уведомления |

**Конфигурация:**
```nginx
# gateway/nginx.conf
events {}

http {
    upstream booking_service {
        server booking-service:8000;
    }
    upstream court_service {
        server court-service:8000;
    }
    upstream payment_service {
        server payment-service:8000;
    }

    server {
        listen 80;
      
        location /api/bookings/ {
            proxy_pass http://booking_service;
        }
        location /api/courts/ {
            proxy_pass http://court_service;
        }
        location /api/payments/ {
            proxy_pass http://payment_service;
        }
    }
}
```

## Вывод

В ходе выполнения лабораторной работы была проведена декомпозиция монолитного приложения на независимые микросервисы, каждый из которых отвечает за свой bounded context. Были выделены четыре сервиса: Booking Service (управление бронированиями), Court Service (площадки и расписание), Payment Service (обработка платежей) и Notification Service (уведомления), что позволило разделить ответственность и упростить дальнейшее развитие системы.
Для обеспечения асинхронного взаимодействия между сервисами был интегрирован RabbitMQ как Event Bus. События BookingCreatedEvent, BookingConfirmedEvent, BookingCancelledEvent публикуются Booking Service и обрабатываются Payment Service (для создания/возврата платежей) и Notification Service (для отправки email/SMS). Payment Service публикует PaymentReceivedEvent, который подтверждает бронирование в Booking Service.
Для внутренней синхронной коммуникации между Booking Service и Court Service используется gRPC, что обеспечивает высокую производительность при проверке доступности слотов и резервировании.
Все сервисы объединены через API Gateway (nginx), который маршрутизирует внешние запросы к соответствующим микросервисам. Инфраструктура описана в Docker Compose, что позволяет развернуть всю систему одной командой.
Полученная архитектура соответствует принципам микросервисной архитектуры: сервисы независимы, масштабируемы, имеют чёткие контракты взаимодействия и изолированные bounded contexts.
